OVERVIEW

Other People Literary Magazine, established in 2019, is known as UC San Diego's first undergraduate-run literary magazine. Among a large team of literary editors and designers, I am currently Other People's lead developer. Therefore, my activities cover a broad range from Web Production to Digital Development to UX Design. I am constantly communicating with directors and other members to accomplish all the tasks needed to maintain and enhance our creative display of UCSD students' works on our website.

Tools: HTML/CSS/JS, GitHub, GitHub Pages (hosting), Jekyll (static site generator), Forestry.io (CMS), Miro (UX wireframes), Webflow (UI layouts), Slack (communication)

Contributions (June - mid-July): Uploaded Collection No. 2 (and Collection No. 3) on the website, code fixes detailed in the changelog, several ADRs in the documentation folder.

INTERNAL MANAGEMENT

I create architectural design records (ADRs) for any major change to the website. These ADRs are about something technical, yet I must write them in a way that the non-developer directors are able to understand. Therefore, I give context on the subject and describe the pros and cons of each decision option. These records document changes that happen in the website code and asserts responsibility onto my actions when making changes. ADRs are both stored in Markdown format within GitHub, as well as inside our Google Drive under our website folder. (See an example of an ADR, in PDF)

Other recording documents include a changelog within the GitHub repository for the minor updates that do not require decisions and a checklist of tasks that I need to complete by certain dates like a collection release.

Other management that has significantly improved the life of the website was code refactoring in HTML, JS, and even files. Before I joined, I knew that the GitHub repository contained a _site folder that was generated in Jekyll, but that folder was not needed when hosting on GitHub Pages. Therefore, it contained a copy of every image file and roughly doubled the size of the repository. This was a bad management of files because the website’s life would not last long if it was growing twice as large. I explained the issue to my directors, who allowed me to remove the _site folder. This refactoring decreased the size of the website from 281 MB to 99 MB, a reduction of 64%. After I uploaded media for Collection No. 2 and 3, the GitHub repository size is 144 MB.

More file issues that I am currently dealing with are writing alt text to add to all images and posts throughout the website and keeping a list of these website files to identify them faster. This spreadsheet shows how I keep track of files inside our Google Drive, on GitHub, and the file size that is recommended to be under 2 MB to increase our GitHub hosting lifetime.

UX Research & Design

Using my previous experiences with UX Design, I fixed many accessibility issues concerning screen-reader and keyboard support. One of the biggest issues was an inaccessible navigation menu in the header. Pressing enter would not open the menu because it was a link instead of a button. Furthermore, links did not have a selection outline and tabbing past the link would fall into the hidden navigation elements when it should tab to the main content. This happened because the navigation container was positioned outside the browser and did not hide the links inside it.

To solve these problems, I changed the navigation link to a button so a user can use the keyboard to trigger the JavaScript function. I included the display: none style to effectively hide the contents of the navigation menu when it was closed. This solved the tabbing logic so the menu button would redirect to the content. These fixes solved the main issues with navigation, but the final issue was the select indicator around the menu button. Searching through the codebase, I identified and removed a user-select: none line that worked against accessibility principles. I understood that was done to secure the published posts from being copied, but it was still easy to select or screenshot the literature and art with that line of code.

Other UX concerns include the search function because the codebase currently utilizes two different navigation menus. Landing pages contain the functional search bar directly in the menu, but posts contain a link to the search page instead of the search bar. The wireframe below details the difference between the two header implementations and the steps it takes to access the search function. This UX issue is still being considered by the directors. In the meantime, I am researching ways that would require a separate search page. Since the menu can be opened in any user approach, accessing this search function in the menu should not be a problem.

Digital Magazine Production

My role has taken over the website production responsibility formerly held by my co-editor-in-chief. Since I have more experience with responsive Markdown/HTML/CSS and the Jekyll site generator, I can quickly curate quality posts for standalone content or a whole Collection. In addition, I use br/, strong, em instead of the Markdown syntax in case the content gets transferred to a different site generator that does not convert Markdown to HTML.

Before I became the Web Producer, the website did not have Collection No. 2 uploaded for about four months. I was able to get all thirteen of the published posts uploaded, formatted, and proofread within eight hours of work! Collection No. 3 has not been released yet, but I already uploaded and formatted the fifteen posts in about three hours. The editorial process is still ongoing, so I would have to proofread the posts later in the final version before our release in Fall!

Even though I have access to the Forestry.io CMS to curate new posts for Collections. I was able to understand the CMS in an hour of reading through onboarding, reading its documentation and creating mock posts, but I prefer working directly in VSCode and Jekyll for efficiency. Forestry is slow with previewing new posts, so I run Jekyll locally to generate the preview within seconds. We use Forestry because it effectively allows the non-developers to create posts and manage the front-matter in a nice UI. The CMS is also headless, meaning that we can switch to a different CMS to connect to our content, if needed.

This example of a post on Forestry shows the poem “Little Doll” by Alexandra Jefson. It is not in Markdown format like most of the other content entries because it was designed to have the text over the illustration background, hence the HTML tags to show two different displays on desktop and on mobile!

New Digital Art Development

The benefit of using GitHub Pages to host our website is that it is entirely custom-coded. Therefore, developers can create anything without being constrained by head-based CMS! This grants us the freedom to develop art within digital media at the limits of our imagination!

Since we are separated by distanced learning, I have not been able to ask everyone for creative ideas to build for the website. However, for Collection No. 3's upcoming release I have implemented a slideshow function for the homepage. The slideshow will showcase featured posts from the latest Collection and link directly to them. My UX design of this slideshow would more effectively indicate that we have a new Collection and makes the Issuu link stand out instead of having it always in the hidden navigation menu. The featured images will fill up the user's screen upon entering the website, so it is easier to see the artwork in comparison to the small image previews in the genre catalogues.

In the upcoming year, when we return to campus, I plan to develop dynamic, native interfaces for digital storytelling and mixed media art. One idea takes the form of a fade-in-out storybook that would give users the chance to interact with the content and keep their focus on paragraphs at a time. The tone of the prose would be amplified by isolated text that pops out to the reader after every click through the dynamic reader.
